# Private single-Household production launch

Family App will launch as a private production app for one Household, not as a public multi-Household product. The database and authorization model should still include `household_id` boundaries so the system can grow later, but the first launch will use a one-time setup path protected by a server-only `FIRST_RUN_SETUP_TOKEN`, invite-only Parent access after that, Household-scoped Child PIN sessions, and no public signup, billing, or generalized onboarding flows. The first production Parent must authenticate with a Supabase magic link before setup; if no Household exists, the verified user can create the Household only by also submitting the setup token, and that setup creates the first Parent row for the authenticated email. Anonymous visitors should only see a locked sign-in screen and should not be able to discover Household names, Parent identities, Child profiles, or whether a PIN exists. Authenticated users whose email is not on the Parent allowlist should see only a generic private-app denial and no Household details. Production launch will start as a fresh Supabase Postgres instance with a complete V1 launch schema, not by migrating demo `localStorage` data. Production release should include V1 slices 1-11 wired to Supabase and server actions, with live Apple Calendar feed fetching the only base feature allowed to follow after release; Calendar screens can still show empty/not-connected states or store the feed URL safely. The `localStorage` demo persistence path should be removed rather than kept as a parallel mode. The migration should proceed in very small vertical slices: create the full schema first, then wire one workflow at a time onto server actions and Drizzle without preserving a second local persistence mode.
